# 技术栈与架构设计

## 1. 技术栈总览

| 层级 | 技术 | 说明 |
|------|------|------|
| 桌面框架 | **Electron** | Node.js 生态操作文件/FFmpeg，Chromium 内核保证 WebGL 兼容 |
| 构建工具 | **electron-vite** + **Vite** | HMR 热更新，极速开发体验 |
| UI 框架 | **React 18** + **TypeScript** | 复杂面板组件化，类型安全 |
| 状态管理 | **Zustand** | 轻量、React 内外皆可用（关键优势） |
| 2D 渲染 | **PixiJS v8** (WebGL) | 高性能图层合成、Mesh 形变、硬件加速 |
| 不可变数据 | **Immer** | 配合 Zustand 实现撤销栈 |
| 拖拽交互 | **@dnd-kit** | 时间线效果块拖拽、素材库拖拽 |
| 测试 | **Vitest** + **Playwright** | 单元测试 + E2E |
| docx 解析 | **mammoth**（主进程） | 海豹骰 Log 富文本提取 |
| 视频编码 | **FFmpeg**（子进程） | 帧序列 → MP4 |
| AI 引擎 | **ComfyUI**（Python 子进程） | 本地生图，JSON API 通信 |

### 为什么不选 Tauri？

个人使用场景下，包体积和内存不是约束。Electron 的 Node.js 生态在与 FFmpeg（管道通信）、ComfyUI（进程管理）、docx 解析等系统级操作上更直接。

### 为什么不混用 Fabric.js/Konva.js？

PixiJS v8 的 EventSystem 足以处理选中、拖拽控制点等交互。两个 Canvas 库共存会引发事件冒泡冲突、坐标系统同步、性能损耗等问题。

### 关键设计决策：Zustand 作为唯一数据桥

```
用户操作 (React)            PixiJS 事件
      │                          │
      ▼                          ▼
┌─────────────────────────────────────┐
│           Zustand Store              │
│  (React 和 PixiJS 共享同一份状态)     │
└─────────────────────────────────────┘
      │                          │
      ▼                          ▼
React 组件自动重绘          PixiJS 订阅回调
(时间线/属性面板)           (Canvas 重新渲染)
```

React 和 PixiJS 是两个独立的渲染世界。Zustand 是它们之间的唯一数据桥梁——两个世界都只读写 Store，不直接通信。

## 3. 核心数据模型

```
Project
├── schemaVersion: "1.0.0"
├── metadata: { name, resolution, fps, createdAt, updatedAt }
│
├── pages: Page[]
│   └── Page
│       ├── id, name
│       ├── layers: {
│       │   background: Layer   ← 底层：漫画框
│       │   character: Layer    ← 中层：角色
│       │   bubble: Layer       ← 顶层：气泡
│       │ }
│       ├── playbackConfig: {
│       │   layerMode: 'sequential' | 'parallel'
│       │   quietHoldMs: number
│       │ }
│       └── logText: string
│
└── presets: AnimationPreset[]
```

**Layer 与 Item 结构：**

```
Layer
├── type: 'background' | 'character' | 'bubble'
└── items: CanvasItem[]
    └── CanvasItem
        ├── id, name
        ├── imagePath: string          ← 素材引用
        ├── baseTransform: {            ← 初始变换
        │   x, y, width, height, rotation, opacity
        │ }
        ├── displayOrder: number        ← 层内排列顺序
        └── timeline: EffectTrack[]     ← 效果轨道列表
            └── EffectTrack
                ├── effectId: string     ← 对应插件 ID
                ├── startTime: number    ← ms
                ├── duration: number|null ← null=持续
                ├── params: object       ← 效果参数
                └── enabled: boolean
```

## 4. 效果插件系统

这是整个应用可扩展性的核心设计。

```typescript
interface EffectPlugin {
  id: string;                    // 唯一标识，如 'move' 'fade' 'scale'
  name: string;                  // 显示名称
  hasDuration: boolean;          // 有结束时间→可拖拽块；无→持续属性
  defaultParams: Record<string, unknown>;

  // 给定进度 0~1，返回变换偏移量
  getTransform(params, progress: number): ItemTransform;

  // 可选：直接操作 PixiJS 对象（材质覆盖等高级效果）
  applyPixiFilter?(item: Container, params, progress: number): void;

  // 参数编辑面板（React 组件）
  ParamsEditor: React.FC<{ params; onChange }>;
}

interface ItemTransform {
  x?: number; y?: number;
  scaleX?: number; scaleY?: number;
  alpha?: number;
  rotation?: number;
  skewX?: number; skewY?: number;
}
```

**添加新效果 = 实现接口 + 注册**，核心引擎无需改动。

**效果叠加**：Compositor 按 EffectTrack 数组顺序依次调用 `getTransform`，累加所有 delta：

```
finalTransform = baseTransform + Σ(effect.getTransform(params, progress))
```

## 5. 渲染管线

每一帧的计算流程：

```
当前时间 T
    │
    ▼
确定当前 Page → PlaybackScheduler 计算每个 Item 的激活时间窗口
    │
    ▼
For each active Item:
    ├─ 获取 baseTransform {x, y, scaleX, scaleY, alpha, rotation}
    ├─ 遍历 timeline 中覆盖当前时刻的效果:
    │    progress = (T - effectStartTime) / effectDuration
    │    delta += effect.getTransform(params, progress)
    ├─ 叠加: finalTransform = baseTransform + Σdelta
    └─ PixiJS 以 finalTransform 渲染 Item
    │
    ▼
按图层顺序合成三层 → 输出帧到屏幕 / 导出缓冲
```

## 7. 项目文件存储格式

每个项目是一个文件夹：

```
MyProject/
├── project.json          ← 项目核心数据
│   {
│     "schemaVersion": "1.0.0",
│     "metadata": { "name": "...", "resolution": {...}, "fps": 24 },
│     "pages": [ ... ],
│     "presets": [ ... ]
│   }
├── assets/               ← 内嵌素材副本
│   ├── char_01.png
│   └── bg_forest.jpg
└── exports/              ← 导出目录
    └── output.mp4
```

- `schemaVersion` 字段支持未来数据迁移
- 素材默认引用外部路径，拖入时才复制到 `assets/`
- JSON 人类可读，git diff 友好
